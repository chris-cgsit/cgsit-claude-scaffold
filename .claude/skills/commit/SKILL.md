# Git Commit

Create a commit for the current changes. If the argument is "push", also push after committing.

## Rules

- **NEVER** add "Co-Authored-By" lines
- **NEVER** add "Generated by" or similar AI attribution
- Commit message format: `module: short description`
- Messages are short, direct, lowercase after module prefix
- Use imperative mood (add, fix, update, remove — not added, fixed)

## Module Prefixes

Use the most specific module affected. Adapt this list to your project's modules.

For cross-cutting changes use the primary module. For build/config-only changes use `build:`.
For documentation or infrastructure use `docs:`.

## Steps

1. Run `git status` and `git diff --staged` (and `git diff` if nothing staged)
2. Analyze the changes
3. If nothing is staged, stage the relevant files (`git add` specific files, not `-A`)
4. Draft a commit message following the format above
5. Show the message to the user, then commit
6. If argument contains "push" or user asked to push: `git push`

## Examples

Good commit messages:
- `docs: add RFC infrastructure`
- `build: initial project scaffolding`
- `build: upgrade Spring Boot to 3.x`
- `service: add transaction validation`

Bad commit messages:
- `Update files` (too vague)
- `Added new validation for the form and updated tests` (too long, no module)
- `fix bug` (no module, no context)
